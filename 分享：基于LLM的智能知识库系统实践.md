# 从零构建一个基于 LLM 的智能知识库系统：实践与思考

> 一个轻量级、零 LangChain 依赖的 AI Wiki 系统，用 FastAPI + LLM API 将非结构化文档自动整理为结构化知识库。

## 项目背景

在日常工作和学习中，我们经常会积累大量的技术文档、学习笔记、项目资料。时间久了，这些资料散落在各个文件夹里，想找的时候找不到，想回顾的时候没有头绪。传统的办法是手动整理——建文件夹、写目录、做索引——但这需要持续的意志力投入，而且知识量大了之后手动维护成本越来越高。

大语言模型的出现带来了新的可能性：如果 AI 不仅能回答问题，还能帮我们自动整理知识呢？

基于这个想法，灵感来源于Andrej Karpathy提出的 **llm-wiki** 的Skill思路，我动手做了这个项目——**LLM Wiki Service**。它是一个后端服务，核心流程是：

```
上传文档 → AI 自动分析 → 创建/合并 Wiki 页面 → 支持 AI 问答
```

项目地址：[https://github.com/zeshawnwang/llm-wiki-service-demo](https://github.com/zeshawnwang/llm-wiki-service-demo) 🌟欢迎 Star！

## 痛点与解法

### 痛点一：知识散落，难以沉淀

**问题**：笔记写了就忘，文档存了就不再看，知识无法形成体系。

**解法**：系统设计了一套"知识摄入流水线"，当上传一篇文档后，AI 会自动分析它和已有知识库的关系，产生五种处理策略：

| 场景 | 说明 | 产出 |
|------|------|------|
| 单主题全新 | 知识库中无相关内容 | 创建 1 个新 Wiki 页面 |
| 多主题全新 | 文档包含多个独立主题 | 创建 N 个新 Wiki + 互相关联 |
| 单主题合并 | 与已有 Wiki 相关 | 更新已有 Wiki（版本递增） |
| 混合策略 | 部分新建 + 部分合并 | 新建 + 更新组合 |
| 内容重复 | 已被充分覆盖 | 跳过 |

这五条策略覆盖了日常知识积累的绝大多数场景：新知识直接建档、补充知识自动合并、重复内容不再浪费 AI 调用费用。

### 痛点二：大模型供应商绑定

**问题**：市面上有 OpenAI、Anthropic、MiniMax 等多种 LLM 供应商，如果代码中硬编码了某家 API，后续切换成本很高。

**解法**：通过统一的配置路由，一行环境变量即可切换供应商：

```bash
LLM_PROVIDER=openai     # 使用 GPT 系列
LLM_PROVIDER=anthropic  # 使用 Claude 系列
LLM_PROVIDER=minimax    # 使用 MiniMax M2 系列
```

所有 LLM 调用都通过 `call_llm()` 一个入口，各供应商的差异封装在各自的 `_call_*` 方法中。新增供应商只需要添加一个配置段 + 一个方法，对上层业务代码零侵入。

### 痛点三：搜索中文内容不准

**问题**：最初的关键词搜索按空格分词，对英文友好但对中文几乎无效——查询"LLM 落地"可能返回零结果。

**解法**：重写了关键词评分算法，支持**字符级中文匹配** + **英文单词匹配**双模式：

```python
# 完整查询匹配（标题命中权重最高）
query_lower in title_lower → +15分

# 中文单字符匹配（适合中文模糊查询）
每个字符命中标题 → +0.5分
每个字符命中标签 → +0.3分
每个字符命中内容 → +0.2分

# 英文单词匹配
query_terms = query_lower.split()
每个单词命中标题 → +10分
```

配合可选的向量语义搜索（通过 LLM 的 embedding API），系统提供 keyword / semantic / hybrid 三种搜索模式。

### 痛点四：LangChain 太重

**问题**：很多 LLM 项目一上来就引入 LangChain，但学习成本高、依赖重、排查问题困难。

**解法**：本项目**零 LangChain 依赖**，直接通过 `httpx` 调用 LLM 的 REST API。这个选择带来了几个好处：

| 对比维度 | 直接调用 API | 使用 LangChain |
|---------|-------------|---------------|
| 代码透明 | 每一行都可见，prompt 可精确控制 | 框架抽象层多，调试需理解内部机制 |
| 依赖体积 | 仅 httpx | LangChain + 几十个子包 |
| 学习成本 | 新人可立刻上手 | 需要学习 Chain/Agent/Tool 等概念 |
| 版本稳定性 | API 变化改一行 URL | 受框架 breaking changes 影响 |
| 启动速度 | 秒级 | 需加载框架模块链 |

当然，这也意味着没有现成的 Agent 框架、流式输出、Memory 管理等能力。但对于"文档 → Wiki → 问答"这个明确场景，直接调用 API 的收益远大于引入框架的成本。项目中还专门写了一个章节分析如果未来需要 LangChain 重构，应该怎么改、改哪些文件。

### 痛点五：日志排查困难

**问题**：早期代码中混杂着 `print()` 和零散的 `logging` 调用，出问题后难以定位。

**解法**：统一了日志模块，所有模块使用相同格式：

```
2026-05-12 15:27:04 - app.services.ai_service - INFO - [OpenAI] 开始调用 - 模型: gpt-4o-mini, temperature: 0.7
2026-05-12 15:27:06 - app.services.ai_service - INFO - [OpenAI] 调用成功 - 模型: gpt-4o-mini
```

每个 LLM 调用都有"开始 → 成功/失败"的三段式日志，配合 `traceback.format_exc()` 的完整堆栈捕获，线上问题定位效率大幅提升。

### 数据存储设计

采用**双层存储架构**，所有数据以 Markdown 文件形式保存在本地磁盘：

- **raw/ 原始资料层**：用户上传的原始文档，AI 只读不修改，保留原始内容作为知识溯源
- **wiki/ 知识层**：AI 自动生成的 Wiki 页面，以 Markdown + YAML frontmatter 格式存储，包含标题、标签、分类、关联页面、版本号等元数据

这种设计的优势：
- **零数据库依赖**：无需安装 MySQL、PostgreSQL 等外部服务，拉起即用
- **可读可迁移**：Markdown 文件人人可读，可直接用 Obsidian、Typora 等工具编辑
- **天然版本管理**：Git 天然适合管理文本文件，每次修改都可追溯
- **简单可靠**：文件系统就是数据库，没有连接池、事务、迁移等问题

### 核心流程：知识摄入流水线

当新文档进入系统时，触发以下完整链路：

```
用户上传文档 → Document Service 存储到 raw/
       ↓
Pipeline 触发 → 读取 raw/ 中未处理的文档
       ↓
AI Service 分析文档主题 → 与已有 Wiki 知识库比对
       ↓
五种处理策略之一：
  ├─ create_new    → 生成新 Wiki 页面 → 写入 wiki/pages/
  ├─ create_multi  → 拆分多主题 → 生成多个新 Wiki + 互相关链
  ├─ merge_into    → 读取已有 Wiki → AI 合并内容 → 版本升级 v1→v2
  ├─ split_merge   → 部分新建 + 部分合并 → 混合处理
  └─ skip          → 内容重复 → 标记跳过
       ↓
重建搜索索引 → 更新知识图谱 → 完成
```

### 问答流程

```
用户提问 → AI Service 接收问题
       ↓
QA Retrieval 模块判断检索策略：
  ├─ ai 模式    → AI 直接浏览 Wiki 目录，自主选择相关页面
  └─ auto 模式  → 页面少时全量给 AI，多时先关键词预筛再 AI 精选
       ↓
AI 阅读选中的 Wiki 页面 → 生成回答
       ↓
返回答案 + 引用来源（附带 Wiki 页面标题和链接）
```

### 设计原则

1. **本地优先**：所有数据存为 `.md` 文件，不依赖外部数据库，`git` 天然可做版本管理
2. **AI 增量融合**：新文档自动判断与已有知识的关系，不做全量重建
3. **配置驱动**：所有开关在 `.env` 中，向量搜索可关闭以节省成本，Git 同步可配置
4. **前后端分离**：后端纯 API + 前端独立项目，可独立部署和扩容

## 一些思考

### 什么时候该选"轻量方案"

做这个项目的过程中，我反复问自己一个问题：**"真的需要 LangChain 吗？"**

答案是否定的。LangChain 的价值在于快速集成多种数据源、编排复杂 Agent 流程、切换不同 LLM 供应商。但如果你像我一样：
- 场景明确（文档 → 知识库 → 问答）
- 调用模式固定（prompt 结构简单、工具数量少）
- 需要精细控制每一个 prompt 字眼
- 希望代码完全透明、可调试

那么直接调用 API 是更好的选择。**选择"够用的简单方案"而不是"强大的复杂方案"，是我在这次开发中最大的收获。**

### AI 的能力边界

在实际测试中，AI 对"新建 vs 合并"的判断准确率令人惊喜——即使是一篇长度适中、主题混合的文档，AI 也能准确识别哪些部分是已有知识的补充、哪些是全新主题。但在以下场景需要人工复核：
- 文档内容过于简短（AI 可能过度判断"新主题"）
- 涉及非常专业的领域术语（AI 可能以为不同术语指向同一概念）
- 知识库中已有大量相似页面（AI 可能选择跳过但实际有差异）

因此，系统设计为**AI 辅助而非 AI 替代**——所有处理结果都可以在 API 层面预览、人工确认后再执行。

## 未来迭代方向

项目当前处于 MVP 阶段，后续计划在以下几个方向持续迭代：

### 1. 前端实现

目前后端 API 已完整可用（可通过 Swagger 文档直接调试），但缺乏直观的图形界面。后续计划使用 React + TypeScript + Vite + Tailwind CSS 构建前端：

- **知识库浏览**：文件树 + Markdown 渲染，支持双向链接跳转
- **知识图谱可视化**：基于 react-force-graph-2d 的力导向图，直观展示页面关联关系
- **导入页面**：拖拽上传文档、运行流水线、查看处理结果
- **问答页面**：Chat 式交互，附带引用来源展示
- **首页仪表盘**：Wiki 数量、文档统计、最近活动

前端设计文档已经完成，开发优先级也已规划好，只待动手。

### 2. Wiki 源文件的云端存储方案

当前所有 Wiki 页面以 Markdown 文件形式存储在本地磁盘，通过 Git 做版本管理。这种方式简单直接，但在以下场景存在局限：

- **多实例部署**：多台服务器需要共享同一份知识库数据，本地文件无法直接同步
- **数据安全**：磁盘故障可能导致数据丢失，本地无自动灾备
- **团队协作**：多人同时操作知识库时，本地文件缺乏并发控制

后续计划引入 **对象存储（S3/MinIO）** 或 **轻量级数据库（SQLite/PostgreSQL）** 作为存储后端，将 Wiki 源文件从本地磁盘解耦，同时保留 Markdown 格式的可读性和可迁移性。

### 3. 热点文件的本地化缓存

当 Wiki 数据量增长到成百上千篇，且存储迁移到远端后，每次读取都从远程拉取会带来性能问题。后续计划实现**热点文件本地缓存**策略：

- **冷热分离**：根据访问频率和最近修改时间，将活跃的 Wiki 页面标记为"热数据"
- **本地 LRU 缓存**：热数据在本地磁盘保留副本，读请求优先命中本地
- **缓存一致性**：远程数据变更时，通过版本号或事件通知机制使本地缓存失效
- **预加载**：根据知识图谱关联关系，预加载当前页面相关的周边页面

这套策略参考了 CDN 的缓存思路，目标是让大规模知识库在远程存储下依然保持毫秒级的访问体验。
